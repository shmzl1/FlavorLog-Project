
# 知味志 FlavorLog 数据库设计文档

本文档是后端与数据库交互的唯一依据。任何表名、字段名、字段类型、外键关系、索引发生变化，都必须先修改本文档，再修改代码。

---

## 1. 数据库基础信息

| 项目     | 值                 |
| -------- | ------------------ |
| 数据库   | PostgreSQL         |
| 推荐版本 | 16                 |
| 数据库名 | flavorlog          |
| 用户名   | flavorlog_user     |
| 密码     | flavorlog_password |
| 默认端口 | 5432               |
| 字符集   | UTF-8              |

连接串：

```text
postgresql+psycopg2://flavorlog_user:flavorlog_password@localhost:5432/flavorlog
```

## 2. 命名规范

### 2.1 表名

统一使用小写复数蛇形命名：

```text
users
food_records
fridge_items
community_posts
```

### 2.2 字段名

统一使用小写蛇形命名：

```text
user_id
created_at
updated_at
food_record_id
```

### 2.3 主键

所有业务表默认使用：

```text
id BIGSERIAL PRIMARY KEY
```

### 2.4 时间字段

所有核心表必须包含：

```text
created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
```

### 2.5 软删除

阶段一不强制软删除。真正删除可以直接 `DELETE`。

如果后续需要软删除，再统一增加：

```text
deleted_at TIMESTAMPTZ
```

# 4. 表结构设计

## 4.1 users 用户表

用途：保存用户账号、登录信息、健康目标和基础画像。

| 字段            | 类型         | 约束                   | 说明                                        |
| --------------- | ------------ | ---------------------- | ------------------------------------------- |
| id              | BIGSERIAL    | PRIMARY KEY            | 用户 ID                                     |
| username        | VARCHAR(50)  | UNIQUE NOT NULL        | 用户名                                      |
| email           | VARCHAR(120) | UNIQUE NOT NULL        | 邮箱                                        |
| password_hash   | VARCHAR(255) | NOT NULL               | 密码哈希                                    |
| nickname        | VARCHAR(50)  |                        | 昵称                                        |
| avatar_url      | TEXT         |                        | 头像地址                                    |
| gender          | VARCHAR(20)  | DEFAULT 'unknown'      | male/female/unknown                         |
| birth_date      | DATE         |                        | 出生日期                                    |
| height_cm       | NUMERIC(5,2) |                        | 身高                                        |
| weight_kg       | NUMERIC(5,2) |                        | 体重                                        |
| health_goal     | VARCHAR(50)  |                        | lose_fat/keep_fit/gain_muscle/control_sugar |
| diet_preference | JSONB        | DEFAULT '[]'           | 饮食偏好                                    |
| allergens       | JSONB        | DEFAULT '[]'           | 过敏原                                      |
| profile_json    | JSONB        | DEFAULT '{}'           | 其他扩展画像                                |
| created_at      | TIMESTAMPTZ  | NOT NULL DEFAULT NOW() | 创建时间                                    |
| updated_at      | TIMESTAMPTZ  | NOT NULL DEFAULT NOW() | 更新时间                                    |

索引：

```sql
CREATE UNIQUE INDEX idx_users_username ON users(username);
CREATE UNIQUE INDEX idx_users_email ON users(email);
```

# 6. JSONB 字段使用约定

## 6.1 允许使用 JSONB 的场景

- 食谱步骤：`steps_json`
- 食材列表：`ingredients_json`
- 营养结构：`nutrition_json`
- AI 原始输出：`result_json`
- 图片 URL 列表：`image_urls`
- 标签列表：`tags`
- 口味向量：`vector_json`

## 6.2 不允许滥用 JSONB 的场景

以下字段必须独立建列，不能塞进 JSONB：

- `user_id`
- `food_record_id`
- `post_id`
- `created_at`
- `updated_at`
- `like_count`
- `comment_count`
- `fork_count`
- 核心查询条件

# 7. 后端访问数据库约定

后端代码分层：

```text
API Router 层
    ↓
Service 业务层
    ↓
Repository / CRUD 数据访问层
    ↓
SQLAlchemy ORM
    ↓
PostgreSQL
```

要求：

1. API 层不直接写 SQL。
2. Service 层不直接拼接 SQL 字符串。
3. 数据库读写统一通过 ORM 或 CRUD 函数。
4. 所有表字段必须和本文档一致。
5. 修改字段前必须先修改本文档和接口文档。
6. 前端不能直接访问数据库，只能访问 FastAPI 接口。
