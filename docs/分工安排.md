本文档用于明确每位成员在项目中的职责边界、对应代码文件、协作方式和验收标准。
所有成员开发前必须先阅读本文档，避免重复开发、文件冲突和接口不一致。

## 2. 总体协作原则

1. 每个人只修改自己负责的文件和目录。
2. 公共文件修改前必须在小组群中说明，并获得组长确认。
3. 接口字段、数据库字段、目录结构不能私自改名。
4. 前端不得直接访问数据库，只能通过后端 API。
5. 后端 API 层不得直接拼接 SQL，必须通过 Service / 数据访问层。
6. 所有接口必须和 `docs/接口文档.md` 保持一致。
7. 所有数据库表结构必须和 `docs/数据库设计.md` 保持一致。
8. 如果实际代码需要改接口或表结构，必须先改文档，再改代码。
9. 每个功能完成后必须能本地运行，并能通过最小联调验证。
10. `main` 分支只放稳定代码，功能开发使用独立分支。

### 9.4 关于社区模块的边界说明

社区模块分为两部分：

第一部分是普通社区 CRUD，包括发帖、评论、点赞、Fork 等基础接口。
第二部分是社交算法，包括口味搭子、相似度匹配、社区推荐、Mock 社区数据。

1. 后端能启动。
2. 数据库能启动。
3. Swagger 能访问。
4. 用户注册登录能跑通。
5. 前端 App 能启动。
6. 前端能连上后端。
7. 饮食记录能新增和查询。
8. 赛博冰箱食材能新增和查询。

在第一优先级完成后继续做：

1. 图片上传。

3. 赛博冰箱生成食谱 Mock。

5. 红黑榜 Mock。
6. 口味搭子 Mock。
7. 社区点赞。

### 11.3 第三优先级

1. 视频极速录入真实算法。

6. Apriori 真实规则挖掘。
7. WebSocket 实时推送。

## 12. Git 分支规范

### 12.1 分支命名

```text
feat/模块名-功能名
fix/模块名-问题名
docs/文档名
chore/配置名
```

示例：

```text
feat/auth-login
feat/food-record-api
feat/fridge-page
feat/recommendation-mock
fix/backend-cors
docs/api-update
chore/update-requirements
```

### 12.2 提交信息规范

```text
feat: 新增功能
fix: 修复问题
docs: 修改文档
style: 修改格式
refactor: 重构代码
test: 添加或修改测试
chore: 修改配置或依赖
db: 修改数据库结构
```

示例：

```text
feat: 完成用户登录接口
feat: 新增饮食记录页面
fix: 修复局域网请求地址错误
docs: 更新分工安排
db: 新增 food_records 表
```

## 13. 接口变更流程

如果需要修改接口：

1. 提出修改原因。
2. 修改 `docs/接口文档.md`。
3. 前端负责人确认字段够用。
4. 后端负责人确认实现可行。
5. 组长确认。
6. 修改代码。
7. Swagger 与接口文档保持一致。
8. 通知所有相关成员同步修改。

未经确认，不允许直接修改接口字段。

---

## 14. 数据库变更流程

如果需要修改数据库表结构：

1. 提出修改原因。
2. 修改 `docs/数据库设计.md`。
3. 修改 `backend/database/init/init.sql`。
4. 修改对应 ORM Model。
5. 修改对应 Schema。
6. 修改对应 Service。
7. 通知前端是否影响接口字段。
8. 组长确认后提交。

未经确认，不允许私自修改字段名、字段类型、外键关系。
